# GFS(Google File System)

## 目标

虽然与以前的分布式文件系统共享许多相同的目标，但我们的设计是由我们的应用程序工作负载和技术环境(包括当前的和预期的)的观察所驱动的，这反映了对一些早期文件系统假设的明显偏离。这促使我们重新审视传统的选择，探索截然不同的设计点。

## 概述

该系统是由许多经常失败的廉价商品组件构建的。它必须不断地监控自己，并在常规的基础上及时发现、容忍和恢复组件故障。系统存储了数量适中的大文件。多GB文件是常见的情况，需要高效管理。小文件必须得到支持，但我们不需要对其进行优化。工作负载主要由两种读段组成：大流式读段和小随机读段。在大的流式读取中，单个操作通常读取数百KB，更常见的是1 MB或更多。来自同一客户机的连续操作通常读取一个文件的相邻区域。一个小的随机读数通常在某个任意偏移量下读取几个KB。工作负载也有许多大的、顺序的写操作，将数据附加到文件中。一旦写入，文件很少被再次修改。支持文件中任意位置的小写，但不必高效。系统必须为并发附加到同一文件的多个客户端高效地实现定义良好的语义。具有最小同步开销的原子性是必不可少的。文件可能在以后被读取，或者消费者可能同时通过文件进行读取。高持续带宽比低延迟更重要。`GFS`提供了一个熟悉的文件系统接口，我们支持创建、删除、打开、关闭、读取和写入文件的常规操作。此外，`GFS`具有快照和记录追加操作。快照以低成本创建文件或目录树的副本。记录追加允许多个客户端同时向同一个文件追加数据，同时保证每个客户端追加数据的原子性。文件被划分为固定大小的块。每个`Chunk i`由主服务器在创建块时分配的不可变且全局唯一的 64 位块句柄标识。块服务器将块存储在本地磁盘上作为`Linux`文件，读取或写入由块句柄和字节范围指定的块数据。它们由主程序独家处理：命名空间锁定保证了原子性和正确性；主操作日志定义了这些操作的全局总顺序。一个文件区域是一致的，如果所有客户端将始终看到相同的数据，无论他们从哪个副本读取。

## 具体实现

每个文件被分成大小为64MB的`chunk`，`GFS`有一个`master`服务器和非常多的`chunk`服务器。`master`服务器维护了多个映射：

1. 文件名到`chunk`句柄数组的映射
2. `chunk`句柄到`chunk`服务器列表的映射(副本机制，为了容错)；

每个`chunk`服务器都在`master服务器`上标记了一个`version`和过期时间`lease expiration`，同时这些服务器被选出一个`primary`服务器，只有`primary`服务器才可以与`master`服务器进行通信。为了保证这些数据在`master`服务器宕机后不丢失，`chunk`句柄数组、和版本号是被持久化到硬盘中的。

每当一个文件被追加了新的`chunk`，或者版本号改变了(指定了新的`primary`)，这意味着`master`服务器必须先在它的日志里追加一条记录，这就限制了`master`服务器写入的速度。这里使用日志而不是数据库的原因是在磁盘上它们有些是用B树或者哈希表来进行存储的，这样对日志进行追加操作的时候会非常高效。

如果`master`服务器崩溃了，必须重构它的状态。我们不需要从首次安装服务器的时间开始重新读取其日志文件。此外，`master`服务器定期通过`checkpoints`将其完整状态写道磁盘，这需要花费一些时间；当`master`服务器重启时，要做的就是回滚到最近的`checkpoints`处，即回滚到该日志中创建这个`checkpoint`的时间点。

### 实际读写

#### 读操作

1. `client`端发送一个读请求，作为读操作，`client`端应用程序要通过一个文件名以及文件名中的一个偏移量(`offset`值)来找到它想读数据的开始点；接着将文件名和偏移量发送给`master`服务器
2. `master`服务器在它的文件表中查找这个文件名，使用偏移量/64MB来找到我们需要的那个`chunk`，然后在`chunk`表中找到保存该`chunk`数据副本的`chunk`服务器列表，并将`chunk`句柄和该列表放返回给`client`
3. `client`端会试着去猜网络中的哪一台服务器离它最近，可能它们就在同一个机架上；接着`client`就会去向该副本服务器发送读请求，同时`client`会将结果进行缓存；当`client`再次去读该`chunk`，那么`client`可能读到的是一个已经给过的`chunk`，可能是其中的1MB的片段或者64KB的片段之类的，我们很可能多次读到同一个`chunk`的不同连续区域
4. `client`端会和其中一个`chunk`服务器进行通信，并告诉它这个`chunk`句柄的偏移量，`chunk`服务器找到这个`chunk`后会将数据返回给`client`

#### 写操作

1. `client`端向`master`服务器发送写请求，表示它想对某个文件进行追加操作，它会去问`master`该文件的最后一个`chunk`文件的位置在哪，因为如果有多个`client`同时对同一个文件进行追加操作，那么`client`可能并不知道这个文件有多长；如果是读操作，可以从任意最新的数据副本中读取，如果是写操作，那就必须对`primary`进行操作，同时`master`服务器需要考虑没有`primary`服务器的情况
    1. No primary: `master`服务器需要去找包含了最新`chunk`副本数据的服务器集合，如果`master`服务器此时没有对应的`primary chunk`，`client`必须等待找出需要的一组有着最近版本`chunk`的`chunk`服务器，`master`服务器从中选择一个来作为`priamry chunk`，其他的`chunk`服务器就作为副本来辅助，与此同时会将最近的版本设定到这些副本服务器中，然后`master`服务器增加版本数字并写入到磁盘中，然后告诉`primary`和辅助副本这个信息，要加上这个版本号，它们也会将版本写入到磁盘中。现在，`master`服务器告诉`client`哪个是`primary`哪个是`secondary`；`master`服务器也会告诉`primary`和`secondary`他们能够去修改这个`chunk`，并给`primary`一个租约或者合约之类的东西，这是一种确保最终不会有两个`primary`的机制。
    2. primary: `client`会发出一份数据拷贝，它希望可以追加到`primary`和所有`secondary`中去，`priary`和`secondary`会将这个数据写入一个临时区域而不是追加到文件，只有当它们拿到了数据，`client`才会发一个消息给`primary`，并表示`primary`和`secondary`都拿到了数据，想把数据追加到文件上。`primary`可能会接受来自大量不同`client`端的并发请求，它会以某种顺序来执行`client`端的请求，一次执行一个。对于每个`client`端发送过来的请求，`primary`会去查看当前`chunk`末尾的偏移值以此来保证在`chunk`内有足够的空间来追加数据。然后将`client`端的记录写入到当前`chunk`的末尾，并且也会告诉所有的`secondary`，将`client`的数据写入到它们`chunk`中相同的偏移量处。如果所有`secondary`确实将数据写入并回复给`primary`，`primary`会回复一个成功的消息给`client`。如果`primary`没得到其中一个`secondary`的响应，`primary`就会回复错误给`client`。如果`client`端从`primary`处得到了一个错误，那么`client`端应该重新发起这个追加操作，并和`master`通信让它找到最后一个`chunk`的所在位置

## 强一致

虽然`GFS`不是强一致性的，但有些思路可以做到。

1. 需要`primary`来检测重复的请求，当第二个请求(`client`端重试)时，`primary`会注意到：实际上之前已经看到了该请求，做或者不做，要尝试确定要不要这个在文件中出现两次
2. 如果需要一个`secondary`(备机)这个角色的话，需要去设计系统，当`primary`告诉`secondary`要做某事时，则`secondary`实际会去做，而不仅仅只是返回错误。对于具有`secondaries`的严格一致性的系统，仅在没有任何补偿的情况下就取消`primary`请求是不行的，`secondaries`必须接受请求并执行，或者`secondaries`具有某种永久性损坏的修复机制来将这个`secondary`退出系统，然后`primary`可以继续处理其余的`secondaries`。`secondaries`必须小心注意不要在`primary`确保所有`secondaries`确实完成执行追加操作之前将数据暴露给进行读操作的`client`。
3. 如果`primary`崩溃了，此时它已经到了最后执行提交的阶段，但是`primary`在确定所有`secondaries`都复制完毕这个操作之前崩溃了。此时`secondaries`中的一个新`primary`将接替老的，剩余的`secondaries`在最近的几次操作中可能会有所不同(因为也许其中一些在老`primary`崩溃之前没有收到消息)，新的`primary`必须从与`secondaries`进行显式重新同步开始，保证它们操作的历史记录的最后时刻是相同的
4. 有时`secondaries`会延时，或`client`可能会从`master`那里得到稍微陈旧的指示，即与某个`secondary`进行通信，系统可能需要通过`primary`发送所有`client`的读操作，因为只有`primary`才可能知道实际发生了哪些操作，或者需要像`primary`一样需要`secondaries`的租用系统，当`secondary`无法合法响应`client`时。

## 缺陷

1. 随着`GFS`的使用量的增加，它们涉及的文件越来越多，`master`存储这些文件的内存将会消耗完毕
2. 对于单个`master`，成千上万的`client`启动并请求`master`会给其造成极大的负载，使其变为系统的平瓶颈
3. 有些应用会发现很难处理奇怪的语义
4. `GFS`的`master`不是那种故障转移自动存储，就像需要人工干预来处理已经永久崩溃的`master`一样需要被更换
